# 产品魔改清单

这份清单用于把当前底座快速改成某个具体 SaaS + AI 产品，同时避免改散核心能力。

核心原则：项目差异优先落在 Product Layer；账号、权限、用户、订阅、钱包、AI 网关、素材、审计、运维这些通用能力保持稳定。

## 1. 开始前先定 8 个值

每次魔改先填完这些值，再动代码。

| 项 | 示例 | 影响文件 |
| --- | --- | --- |
| 产品代码 | `media_studio_ai` | `admin/src/product/config.ts` |
| 产品名称 | `灵感影像工坊` | `admin/src/product/config.ts` |
| 默认接入名称 | `灵感影像工坊 API 接入` | `admin/src/product/access.ts`, `backend/src/modules/product/defaults.rs` |
| 默认接入 slug | `media-studio-default` | `admin/src/product/access.ts`, `backend/src/modules/product/defaults.rs` |
| 默认套餐 | `studio` | `admin/src/product/subscription.ts`, `backend/src/modules/product/defaults.rs` |
| 套餐功能标记 | `image:generate`, `video:generate`, `audio:generate`, `asset:library` | `admin/src/product/subscription.ts`, `backend/src/modules/product/defaults.rs` |
| 服务端 Key scopes | `ai:invoke` | `admin/src/product/access.ts`, `backend/src/modules/product/defaults.rs` |
| AI 计费默认文案 | `影像生成余额充值`, `生成异常人工退款` | `admin/src/product/ai.ts`, `backend/src/modules/product/defaults.rs` |

`default application slug` 是初始化幂等识别的关键字段，确定后不要随意改。已经上线后改 slug，等同于告诉系统“这是另一个默认接入配置”。

## 2. 前端改动点

优先只改 `admin/src/product/`。

| 文件 | 改什么 | 不该做什么 |
| --- | --- | --- |
| `config.ts` | 品牌、标题、登录/仪表盘副标题 | 不在这里放接口地址或密钥 |
| `access.ts` | 默认接入名称、slug、认证模式、心跳、席位、Server Key scopes | 不扩大到未被后端支持的 scope |
| `subscription.ts` | 套餐代码、显示名、默认席位、功能标记 | 不把真实支付价格写死在这里，价格应走计费/商品模块 |
| `ai.ts` | 钱包调整快捷金额、操作原因、模型模板覆盖 | 不保存供应商密钥 |
| `menu.tsx` | 产品专属菜单入口 | 不改核心 SaaS 菜单来挂产品页 |
| `routes.tsx` | 产品专属页面路由 | 不绕过受保护路由和权限 |
| `ProductSetupPage.tsx` | 只有初始化体验变化时才改 | 不在页面里硬编码业务写库逻辑 |

前端改完后先打开：

```text
/product/setup
```

检查：

- 初始化对象是否符合项目；
- dry-run 计划是否符合项目；
- preflight 是否有 `conflict`；
- 执行历史是否能追溯资源 ID。

## 3. 后端改动点

优先只改 `backend/src/modules/product/`。

| 文件 | 改什么 | 触发条件 |
| --- | --- | --- |
| `defaults.rs` | 后端 Product 默认值 | 每个项目基本都会改 |
| `seed_plan.rs` | dry-run 步骤和 warnings | 初始化步骤发生变化 |
| `preflight.rs` | 只读检查当前租户状态 | 新增需要检查的数据库资源 |
| `execute.rs` | 受控写库初始化 | 新增可幂等创建的资源 |
| `history.rs` | 初始化执行历史摘要 | 执行审计结构变化 |
| `admin.rs` | Product 后台接口 | 新增 Product 专属 API |
| `mod.rs` | 模块导出 | 新增 Product 子模块 |

后端 Product Layer 可以查询核心表，但不要把项目特有规则塞回核心模块。确实需要通用能力时，再评估是否提升到核心模块。

## 4. 初始化规则

当前初始化链路分四段：

1. `GET /api/admin/product/seed-plan`：返回 dry-run 计划和 preflight，只读。
2. `POST /api/admin/product/seed-plan/execute`：受控执行，只创建缺失的默认接入配置和服务端 Key。
3. `GET /api/admin/product/seed-plan/executions`：读取初始化执行历史，不返回历史密钥明文。
4. `/product/setup`：前端统一入口。

执行接口的边界：

- 必须传 `confirm: "INITIALIZE_PRODUCT"`。
- 必须有 `system:update`、`app:create`、`server_api_key:update`。
- preflight 有 `conflict` 时拒绝执行。
- `app_secret` 和 `plain_key` 只在新建时返回一次。
- 套餐预设目前没有目录表，只能人工核对。
- AI 钱包/任务操作原因来自 Product defaults，不写库。

## 5. 新增产品专属资源时怎么做

如果某个项目需要额外初始化资源，按这个顺序加：

1. 在 `defaults.rs` 增加默认值。
2. 在 `seed_plan.rs` 增加 dry-run step。
3. 在 `preflight.rs` 增加只读检查，先能判断 `exists/missing/conflict/manual/config_only`。
4. 在 `execute.rs` 增加写库逻辑，只允许幂等创建，冲突必须拒绝。
5. 在 `history.rs` 增加安全摘要，不输出密钥或敏感配置。
6. 更新 `../architecture/API接口文档.md` 和 `backend/openapi.yaml`。
7. 更新 `/product/setup` 展示。
8. 跑后端测试和前端构建。

如果新增资源需要新表，先停下来做数据库设计和 migration，不要把结构化业务数据塞进 `metadata` 凑合。

## 6. 验证命令

后端：

```powershell
cargo fmt --manifest-path backend\Cargo.toml
cargo test --manifest-path backend\Cargo.toml
```

前端：

```powershell
npm run lint
npm run build
```

如果改了路由，`cargo test` 会检查 `backend/openapi.yaml` 是否列出所有注册路由。

## 7. 不要碰的区域

除非明确要扩展核心能力，否则不要为了某个项目去改：

- `auth`
- `iam`
- `customer`
- `subscription`
- `ai` 网关/钱包/任务核心结算
- `server_api`
- `audit`
- `system`
- 旧兼容区：`license`、`device`、`release`、`secure_script`、`client-sdk`

旧兼容区可以继续维护 bug，但不要作为新 SaaS + AI 项目的主线扩展点。

## 8. 交付检查

魔改完成后至少确认：

- `/product/setup` dry-run 和前端快照一致。
- preflight 没有 `conflict`。
- 执行初始化后，历史记录能看到创建/复用的资源 ID。
- 新建密钥明文只在本次结果中出现。
- `../architecture/API接口文档.md` 已同步。
- `cargo test`、`npm run lint`、`npm run build` 通过。
